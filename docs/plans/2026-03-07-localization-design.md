# MacMover Localization Design

## Goal

Add end-to-end English and Korean localization to the macOS app without changing the meaning of the manifest contract or weakening the current `swift build` / `swift test` workflow.

## Scope

### In Scope

- SwiftUI shell text
- tab labels, card titles, buttons, placeholders, empty states
- `AppState` runtime status messages
- Markdown report headings, section labels, empty-state text, and count-based summaries
- user-facing titles/details/reasons/actions generated by core, exporter, and importer services

### Out of Scope

- translating manifest field names, schema keys, enum raw values, report filenames, or bundle layout paths
- adding an in-app language picker in v1
- re-translating report files that were already generated in an older export bundle

## Current State

User-facing strings are currently spread across multiple modules:

- `Sources/App/*.swift` for SwiftUI text and status presentation
- `Sources/Reporting/MarkdownReportWriter.swift` for report scaffolding
- `Sources/Core/*.swift`, `Sources/Exporters/*.swift`, and `Sources/Importers/*.swift` for `PreflightCheck`, `ManualTask`, and `StepResult` text

This means a UI-only localization approach is insufficient. A workable design must cover both view code and non-UI string generation.

## Approaches Considered

### Approach A: Per-target String Catalogs

Put `.xcstrings` files into `App`, `Reporting`, and other modules, then localize strings at each call site with `Text("...")` or `String(localized:)`.

Pros:

- modern Apple workflow
- good Xcode authoring experience
- easy SwiftUI adoption in the main app target

Cons:

- key ownership becomes fragmented across targets
- non-UI text still needs explicit bundle handling
- current SwiftPM + `swift build` verification flow has more tooling risk than plain `.strings`

### Approach B: Dedicated `Localization` Module With `.strings` / `.stringsdict`

Create a small SwiftPM target that owns the localization resources and exposes typed helper APIs used by `App`, `Reporting`, `Core`, `Exporters`, and `Importers`.

Pros:

- one keyspace for the whole product
- stable with SwiftPM CLI workflows
- easy to reuse from UI and non-UI modules
- straightforward pluralization support
- smaller migration surface than redesigning shared models

Cons:

- slightly more manual than pure SwiftUI string extraction
- call sites need a typed helper instead of relying on raw inline strings

### Approach C: Custom JSON-based Translation System

Load app translations from JSON and resolve keys with custom runtime code.

Pros:

- maximum control

Cons:

- duplicates Apple localization features
- more parsing and formatting code
- unnecessary maintenance cost for two languages

## Recommendation

Use **Approach B** for the first implementation:

- add a dedicated `Localization` module
- store translations in `en.lproj/Localizable.strings`, `ko.lproj/Localizable.strings`, and `Localizable.stringsdict`
- expose a typed `L10n` API for shared use across all modules

This is the best fit for the current repository because the app already depends on SwiftPM and validates with `swift build` and `swift test`. The localization layer needs to work from SwiftUI views, report writers, and core workflow code, not only from the app target.

## Recommended Architecture

### 1. Add a Dedicated Localization Target

Create:

- `Sources/Localization/Localization.swift`
- `Sources/Localization/Resources/en.lproj/Localizable.strings`
- `Sources/Localization/Resources/ko.lproj/Localizable.strings`
- `Sources/Localization/Resources/en.lproj/Localizable.stringsdict`
- `Sources/Localization/Resources/ko.lproj/Localizable.stringsdict`

Package changes:

- add `defaultLocalization: "en"` to `Package.swift`
- add a new library target: `Localization`
- make `App`, `Reporting`, `Core`, `Exporters`, and `Importers` depend on `Localization`

App bundle changes:

- add `CFBundleDevelopmentRegion = en`
- add `CFBundleAllowMixedLocalizations = YES`
- add `CFBundleLocalizations = [en, ko]`

Apply the plist changes to both:

- `Sources/App/Info.plist`
- `Xcode/App-Info.plist`

This keeps package resources usable from the app even though the localized resources live outside the main bundle.

### 2. Use Typed Localization Keys

Add a small wrapper API instead of raw string literals:

```swift
public enum L10nKey: String {
    case appTabOverview = "app.tab.overview"
    case statusExportRunning = "status.export.running"
    case reportWorkspaceScanTitle = "report.workspace_scan.title"
}

public enum L10n {
    public static func string(_ key: L10nKey, locale: Locale = .autoupdatingCurrent) -> String {
        String(
            localized: String.LocalizationValue(key.rawValue),
            bundle: .module,
            locale: locale
        )
    }

    public static func format(
        _ key: L10nKey,
        locale: Locale = .autoupdatingCurrent,
        _ arguments: CVarArg...
    ) -> String {
        let format = string(key, locale: locale)
        return String(format: format, locale: locale, arguments: arguments)
    }
}
```

This keeps call sites consistent and prevents key drift across modules.

### 3. Treat UI Text and Workflow Text the Same Way

The same localization layer should supply:

- tab labels in `ContentView`
- card titles and button labels in all tabs
- `AppState` status text and preview summaries
- `MarkdownReportWriter` headings and section names
- workflow-generated strings inside `PreflightService`, `ManualTaskEngine`, `VerifyEngine`, `ExportCoordinator`, `ImportCoordinator`, `HomebrewExporter`, `VSCodeExporter`, `GitGlobalExporter`, `WorkspaceApplyCoordinator`, and `WorkspacePromoteCoordinator`

This avoids the current split where UI strings and core workflow strings would otherwise diverge.

### 4. Support Plurals Up Front

Some strings should not be implemented with string interpolation only:

- `1 tool detected` / `N tools detected`
- `N drift items`
- `N files exported`
- `Ready items: N`
- `N entries`

Use `.stringsdict` for these keys from the start. English and Korean have different plural behavior, and this avoids rewriting the same call sites later.

## Locale Behavior

### v1 Recommendation

Use the macOS preferred language automatically. Do not add an in-app language picker in the first pass.

Why:

- smaller diff
- no settings persistence needed
- no additional environment plumbing through SwiftUI
- good enough for an English/Korean first rollout

### Future Extension

If a manual language selector becomes necessary later, add an app-level locale override in `AppState` and route `L10n` through an injected locale provider instead of only `Locale.autoupdatingCurrent`.

## Persisted Artifact Behavior

This needs to be explicit:

- reports generated during export/import/verify are localized at generation time
- `BundlePreviewService` only reads already-written markdown, so older bundle reports are shown in the language they were generated in
- manifest `item.title` and `manualTasks` text are persisted as plain strings today, so imported bundles can contain source-language text

This is acceptable for the first localization pass because it avoids a manifest schema change. If product requirements later demand fully re-localizable bundles across languages, the manifest model will need semantic message identifiers and parameters instead of only rendered strings.

## Error Handling Strategy

Current status messages often append `error.localizedDescription`. To localize these consistently:

- make `CoreError` conform to `LocalizedError`
- localize user-facing error descriptions in `Sources/Core/ErrorTypes.swift`
- keep raw stderr, file paths, and command outputs unlocalized where they are diagnostic payload rather than UI copy

This preserves useful debugging data while localizing the framing text around it.

## Testing Strategy

### Unit Tests

Add direct tests for the localization layer:

- English key lookup returns expected English text
- Korean key lookup returns expected Korean text
- pluralized strings render correctly for both locales

### Reporting Tests

Update report tests to render with explicit locale arguments:

- English output still contains expected headings
- Korean output uses translated section headings

### App Tests

Update `AppState` and file-content tests so they stop asserting raw English string literals in source files. Prefer behavior assertions over source-code string matching.

## Migration Plan

### Phase 1

- add `Localization` target and resources
- wire package and plist configuration
- localize app shell text and `AppState` statuses

### Phase 2

- localize report writer output
- localize preflight/manual task/export/import/verify messages

### Phase 3

- update tests and README
- verify `swift build`, `swift test`, and Xcode project registration

## Risks

### SwiftPM Resource Wiring

The main risk is package resource visibility across `swift build` and Xcode app runs. The design above reduces this risk by using plain localized resource bundles and explicit host-app localization metadata.

### Snapshot Artifact Language Mixing

Older bundles will still show source-language report content. This is a product behavior, not a bug, and should be documented.

### Test Fragility

Current tests that search for raw English text in source files will become brittle once strings move behind localization helpers. Those tests should be rewritten before claiming localization is complete.

## Summary

For this repository, the pragmatic v1 solution is:

1. central `Localization` module
2. package-backed `.strings` / `.stringsdict` resources
3. system-language selection only
4. localized UI, statuses, reports, and workflow text
5. no schema change in the first pass, while explicitly accepting source-language bundle artifacts
